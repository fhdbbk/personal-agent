# 2026-05-10 — token/sec stats in the UI

## Goal

Pick up an abandoned thread from a previous session: surface per-turn token throughput (tokens/sec, eval tokens, wall time) under each assistant message. Ollama already returns `eval_count` / `eval_duration` on the final streaming chunk; the loop just wasn't reading them. Mid-session, also added **TTFT** (time-to-first-token) to the same payload — a single number that tells you whether a slow turn is "model loading" or "model generating", which is the first thing you want to know on a CPU-bound laptop project.

## What we did

1. **Backend — collect stats inside `run_turn`.** [backend/app/agent/loop.py](../../backend/app/agent/loop.py) now keeps a `last_chunk` reference inside the per-iteration `async for chunk in stream` and, after the stream drains, reads `eval_count` / `eval_duration` / `prompt_eval_count` off it via `getattr` (so a model that omits any of them doesn't crash the loop). Counters are summed across iterations (`total_eval_tokens`, `total_eval_ns`, `total_prompt_tokens`, `model_calls`) so a multi-step ReAct turn reports the *whole* turn's compute, not just the final reply step.
2. **Changed `run_turn`'s return shape from `str` to `tuple[str, dict]`.** The dict is built by a new `_make_stats(...)` helper: `eval_tokens`, `prompt_tokens`, `eval_seconds` (rounded to 0.01), `tokens_per_sec` (rounded to 0.1), `model_calls`. The docstring now says "Returns `(final_text, stats)`" and explains why a multi-step turn hits the model N times. No other callers needed updating — `run_turn` only has the one caller in `chat_stream`.
3. **WS frame — `done` carries `stats`.** [backend/app/api/chat.py](../../backend/app/api/chat.py) unpacks `final, stats = await run_turn(...)`, then sends `{"type": "done", "conversation_id": ..., "stats": {...}}`. Updated the protocol-shape comment at the top of `chat_stream` so the frame contract is documented in-place. Also added `tps=%.1f` to the per-turn server log line.
4. **Frontend — render under the assistant message.** [frontend/src/App.tsx](../../frontend/src/App.tsx) gained a `TurnStats` interface, an optional `stats?: TurnStats` field on `message`-kind transcript items, an updated `DoneFrame` type, and the `done`-frame handler now walks the transcript backwards to attach stats to the most recent assistant message (covers the multi-step case where tool cards are interleaved between the assistant text and the `done` frame). New `<StatsLine>` component renders `5.6 tok/s · 14 tokens · 2.5s · prompt 1115`, with optional `· N calls` when the turn was multi-step.
5. **Styling.** Added a `.stats` class in [frontend/src/App.css](../../frontend/src/App.css) — monospaced, 11px, muted grey. Sits flush under the message content.
6. **Smoke updated.** Modified [scripts/smoke_chat_ws.py](../../scripts/smoke_chat_ws.py) to print the stats payload on `done`. With the server running, a single-turn count-to-five message produced: `[stats] {'eval_tokens': 14, 'prompt_tokens': 1115, 'eval_seconds': 2.5, 'tokens_per_sec': 5.6, 'model_calls': 1}`. Tool-using turns also work — re-ran [scripts/smoke_agent.py](../../scripts/smoke_agent.py) (read_file → write_file approve → write_file deny), no AgentError, no shape mismatch on the new `(final, stats)` unpack. Frontend `tsc --noEmit` passed clean.
7. **TTFT extension (same session, second pass).** Added `ttft_seconds` to the stats payload. In [`loop.py`](../../backend/app/agent/loop.py) we capture `t_start = time.perf_counter()` at the top of `run_turn` and `t_first_token` the first time any iteration's stream emits non-empty `msg.content`. Compute `ttft = t_first_token - t_start` at exit and pass it through `_make_stats`. `None` if the turn produced no text (e.g. tool-only iterations followed by an error). Frontend renders it as `· ttft 0.42s` between `eval_seconds` and the optional `model_calls`. WS smoke after the change reported `ttft_seconds: 42.94` on a cold model-load turn — exactly what we want this number to surface, since the same turn's `eval_seconds` was only 1.6s. With TTFT split out it's instantly clear *where* the latency is.

## Decisions made

- **No ADR.** This is a small UX addition built on top of values Ollama already emits; nothing changed in the architecture, transport, or tool protocol. The shape change to `run_turn`'s return value is internal — the only caller was updated in the same commit. If we ever expose the stats *outside* the WS (e.g. as a Prometheus metric) that would deserve a note.
- **`prompt_tokens` is summed across model calls, not "prompt size of the last call."** This makes the displayed number a compute-cost proxy (total prompt tokens evaluated this turn) rather than a "how big is the context" indicator. Felt closer to what a user would want when watching cost. The `· N calls` suffix on multi-step turns is the hint that the prompt number is summed.
- **Round at the backend, not the UI.** `eval_seconds` and `ttft_seconds` to 2dp, `tokens_per_sec` to 1dp. This way the stats payload is print-friendly for smoke-script logging and we don't sprinkle `.toFixed(...)` calls all over the frontend (the one in `<StatsLine>` is decorative — appending the unit cleanly).
- **TTFT lives in the loop, not the WS handler.** Two reasonable places: (a) `chat_stream` could timestamp the first `token` event it forwards to the WS, or (b) `run_turn` could timestamp the first content delta it observes from Ollama. We picked (b). Two reasons: it's strictly more accurate (no async-event-bubbling delay between loop and WS handler), and it keeps the stats payload entirely a function of one place — the loop — so the WS layer doesn't need its own timer plumbing alongside `t0`. Difference between (a) and (b) is sub-millisecond in practice but the cleaner ownership matters more.
- **`ttft_seconds: null` for "no text emitted."** Distinguishes "we never got a first token" from "we got it instantly." Frontend treats `null` as "don't render the field." Could've used 0 or -1; null was the most honest type signal and `JSON.stringify` round-trips it cleanly.

## Snags + fixes

- **`eval_count` lives on the chunk, not on `chunk.message`.** First read of the Ollama Python lib's `ChatResponse` shape — the per-chunk fields `eval_count`, `eval_duration`, `prompt_eval_count`, `total_duration` are top-level on the chunk object, alongside `done` / `done_reason`. They're only populated on the final `done=True` chunk. Hence the `last_chunk = chunk` capture inside the stream loop and the `getattr(last_chunk, ...)` reads after.
- **No browser eyeball this session.** Verified the wire format and the frontend typechecks; did not run Vite + browse the UI (running headless WSL). Carrying forward as a manual check for next session — the rendering is straightforward enough that I'd be very surprised if it broke, but per [CLAUDE.md](../../CLAUDE.md) frontend-conventions we should still eyeball it.
