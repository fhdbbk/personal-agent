2. **UI papercuts pass** (`d54caee`). Three small fixes after eyeballing in the browser:
   - `.tool-args` / `.tool-result` get `overflow-wrap: anywhere`, so long URLs (notably fetch_url args) wrap inside the card instead of producing a horizontal scrollbar.
   - A pulsing `thinking…` row at the bottom of the transcript when `busy=true`, no assistant message is actively streaming, and no tool is awaiting approval. Covers two gaps: between user-send and the first token, and between `tool_result` and the next iteration's first token. New `shouldShowThinking(transcript)` helper encodes the rule. Same `pulse` keyframe is applied to the `running…` status on tool cards.
   - Removed the now-dead `{it.content || (busy ? '…' : '')}` placeholder inside `<div class="content">` — the backend only emits `token` frames with non-empty deltas, so the assistant message is never created with empty content.
3. **Clarified the `prompt_tokens` UI tooltip** (same commit). Fahad spotted that `prompt 3003` (on a 2-call web_search turn) was followed by `prompt 1273` on the next turn — the *opposite* direction from what you'd expect if you were reading the number as "context size." It isn't: per the 2026-05-10 token-stats decision, `prompt_tokens` is *summed across model calls in the turn*, so a single-call conversational turn can show fewer prompt tokens than a multi-call tool turn even as the buffer grows. The `· N calls` suffix was the design's tell, but easy to miss. Added `<span title="...">` tooltips on the three ambiguous segments — `ttft`, `N calls`, and especially `prompt N` ("sum of prompt tokens across every model call this turn — compute cost, not context size. tool results inflate later calls but are not persisted.").
4. **Configurable context window** (`1453c8d`). Ollama's default `num_ctx=4096` is too small once tool results enter the conversation — one fetch_url result can blow the budget on its own. New `PA_OLLAMA_NUM_CTX` setting defaults to `32768` (qwen2.5/3's native training window, our safety ceiling on a 16 GB laptop). Set higher via `.env` to opt in to YaRN-extended context (quality risk past native window). Renamed `device_options()` → `ollama_options()` since it now packages both `num_gpu` (from device) and `num_ctx`. Function always returns a non-empty dict, so callers in [backend/app/agent/loop.py](../../backend/app/agent/loop.py) and [backend/app/api/chat.py](../../backend/app/api/chat.py) drop the `extra = {"options": options} if options else {}` splat-pattern and pass `options=ollama_options()` directly.
5. **Verified the new context took effect**: `ollama stop qwen3.5:4b` → ran `scripts/smoke_chat_ws.py` to trigger a fresh load → `ollama ps` now shows `CONTEXT 32768, SIZE 6.8 GB` (was 4096 / 6.2 GB; the ~600 MB delta is the KV cache, well under the worst-case ~2 GB because Ollama doesn't pre-allocate the full buffer).
6. **Agent-loop unit tests** (`bd4d585`, `46d6455`). First real test suite for [backend/app/agent/loop.py](../../backend/app/agent/loop.py). 11 tests, all pass in ~0.5s, no live Ollama needed. The loop's transport-agnostic shape (Ollama via `AsyncClient`, UI via `on_event` + `request_approval` callbacks) made this clean to fake. Coverage:
   - Single-call happy path — content + stats wire-up, token frames emitted.
   - Multi-iteration stats aggregation — `eval_tokens`, `eval_ns`, `prompt_tokens` summed across calls; `model_calls` counted.
   - Tool dispatch — `tool_call` / `tool_result` frames emitted; tool result appended as `{"role": "tool", "content": ...}` to the next call's `messages`.
   - Unknown tool name — `ok=False` with "unknown tool" body fed back to the model.
   - Wrong/missing kwargs — `TypeError` caught, `ok=False` with "argument error: ..." (distinct branch from unknown-tool).
   - Approval approved → tool runs. Approval denied → tool body does **not** run, `ok=True` (not an error) with "User denied this action." body, retry counter not ticked.
   - Repeated same-tool failure → `AgentError` on the 3rd consecutive failure (default `max_retries_per_tool=2`).
   - Error counter resets when a different tool fails — keyed to the tool name, not total errors.
   - `PA_AGENT_MAX_STEPS=2` → loop aborts via `AgentError("MAX_STEPS")` when the model never returns a final answer.
   - Options threading — every `client.chat(...)` call gets `stream=True`, `options.num_ctx`, `tools`, and the messages list.
7. **Pytest infra.** Added `pytest>=9.0.3` and `pytest-asyncio>=1.3.0` under a new `[dependency-groups] dev` section. `[tool.pytest.ini_options]` sets `pythonpath=["."]` (project still isn't installed as a package — see CLAUDE.md conventions), `testpaths=["backend/tests"]`, and `asyncio_mode="auto"` so async tests don't need a decorator. AI.md commands section grew a `uv run pytest` line and the repo layout now lists `backend/tests/`.

## Decisions made

- **No ADR.** Nothing in today's work changed the architecture, transport, or tool protocol. `num_ctx` is a single config knob, the UI changes are cosmetic + one tooltip clarification, the test infra is internal scaffolding. The decision *not* to switch device off CPU (next bullet) is a personal preference, recorded as memory rather than as a project ADR.
- **CPU inference is intentional.** During the num_ctx work, `ollama ps` showed the model on CPU after our backend forced a reload (it had previously been on GPU because something else had loaded it first). When asked, Fahad confirmed `PA_OLLAMA_DEVICE=cpu` is the deliberate config — this whole project is studying CPU inference (Phase 6 even profiles/replaces hot paths in native code), so slow ttft on a cold load is the expected baseline, not a problem to solve. Saved as a feedback memory so future sessions don't keep nudging toward GPU. The trade-off is felt: cold-load ttft was ~46s for a "count to five" turn.
- **`prompt_tokens` semantics stay summed; only the tooltip changes.** Fahad's confusion was legitimate but the alternative semantics ("last call's prompt size as context proxy") would lose the compute-cost reading on multi-call turns. A tooltip is cheaper than a label change and preserves the original design intent. The session-log decision from 2026-05-10 stands, just now self-documenting in the UI.
- **`num_ctx=32768` is the conservative default, not 65536.** Fahad initially asked for 64k; I flagged the VRAM cost (~2 GB KV cache extra) and the YaRN-extension quality risk past qwen's native 32k window. He chose "make it configurable, default 32k" — so the safe ceiling is the default and the .env knob is there if a future turn needs more headroom.
- **Renamed `device_options()` → `ollama_options()`.** Two reasons: (a) the function now does more than device (returns `num_ctx` too), (b) it always returns a non-empty dict so the awkward `extra = {"options": options} if options else {}` splat at the call sites goes away. Two callers, two small edits. Cheap rename, clearer name.
- **Pytest config lives in `pyproject.toml`, not `pytest.ini` or `conftest.py`.** One file fewer in the root. `pythonpath=["."]` mirrors the `sys.path` shim the smoke scripts use, so we don't have to install the project as a package yet — CLAUDE.md still notes that as the trigger for switching to a real build config.

## Snags + fixes

- **`AI.md` modified by uncommitted prior-session work.** `git status` showed `AI.md` modified even before today started. Turned out fine — the changes were the table-row + repo-layout + commands updates that came with `list_files`/`fetch_url`/ADR-0006. Got swept into the right commit (`7902a7e`). Worth noting: `CLAUDE.md` is a symlink to `AI.md`, so we only ever edit `AI.md`.
- **Backend not running with `--reload`.** I started it as `uvicorn ... --port 8000` (no `--reload`), so the first num_ctx edit didn't take effect until I killed + restarted with the flag. Should default to `--reload` for dev. The CLAUDE.md commands example already uses `--reload`, so this was just me skipping it.
- **`ollama ps` showed GPU after restart.** Cause: Ollama keeps loaded models cached and reuses them across clients. Whatever loaded the model first (likely Open WebUI or a stray `ollama run`) won. Our `num_gpu=0` only takes effect when Ollama actually has to load. Forcing `ollama stop qwen3.5:4b` and re-running our smoke gave us a fresh load with our options. Documenting in case future-me forgets: changing `num_ctx` or `num_gpu` in `.env` doesn't reload an already-loaded model — `ollama stop <model>` first.
