# 2026-05-03 (pm) — Warm-up loose ends + ADR 0003 for the agent loop

## Goal

Clear two pending papercuts from the Phase 1 backlog (`request_timeout_s` not applied; no buffer reset), then design Phase 2 on paper — write the ADR before any agent-loop code lands.

## What we did

1. **Applied `request_timeout_s` to Ollama calls.** [backend/app/api/chat.py](../../backend/app/api/chat.py) — passed `timeout=settings.request_timeout_s` into `AsyncClient(...)`. Ollama forwards kwargs to `httpx.AsyncClient`, so the read-timeout naturally becomes a per-chunk idle timeout for streams (long generations are fine, stalled connections abort). One-line fix once the right knob was found; the alternative — `asyncio.wait_for` per chunk — would have been more code for the same behavior.
2. **Added `POST /chat/reset`.** [backend/app/api/chat.py](../../backend/app/api/chat.py) — new `ResetRequest`/`ResetResponse` Pydantic models and a route that calls `buffer.clear(conversation_id)`. The buffer's `clear()` already existed.
3. **Frontend "New" button.** [frontend/src/App.tsx](../../frontend/src/App.tsx) — moved `conversation_id` from a `useRef` to `useState` so the header re-renders after a reset. New `newChat()` async handler: clear messages locally, rotate the cid, fire-and-forget `POST /chat/reset` for the *old* cid (best-effort eviction; the rotated cid alone is enough from the model's perspective). Disabled while busy. Added matching CSS.
4. **Smoke tested end-to-end.** `POST /chat` two-turn conversation: turn 1 set "favorite color is teal", turn 2 recalled "Teal". After `POST /chat/reset`, the same recall question returned "Blue" (model default) — buffer cleared as expected. WS streaming smoke (`smoke_chat_ws.py`) still passes (ttft ~730 ms warm). `npx tsc --noEmit` on the frontend is clean.
5. **Wrote [ADR 0003](../decisions/0003-agent-loop.md).** Pinned Phase 2 design before code:
   - **Native Ollama tool-calling**, not prompt-and-parse JSON. Ollama's Python client takes `tools=[...]` and returns structured `tool_calls`; Qwen3 is trained for it. The educational lift is the loop, not the parser.
   - **Sequential ReAct** loop, one tool at a time, `MAX_STEPS=8` (env `PA_AGENT_MAX_STEPS`). On tool error, feed the error string back as the tool result for up to 2 retries, then fail the turn.
   - **Tool registry is a dict** of `Tool(fn, schema, requires_approval)` dataclasses. No decorators, no auto-discovery.
   - **Wire protocol extends ADR 0002.** New transcript frames `tool_call` / `tool_result`, plus a server→client `tool_approval` request that pauses the loop on a future until the client replies with `approval_response`. ADR 0002's bidirectional WS choice carries this for free.
   - **Sandboxing.** `read_file`/`write_file` scoped to `PA_AGENT_SANDBOX` (default `./sandbox/`); `write_file` and `python_exec` require approval (override via `PA_AGENT_AUTO_APPROVE` for dev); `python_exec` runs in a subprocess with `RLIMIT_CPU` + `RLIMIT_AS` and is explicitly *not* hardened against an adversarial model — it's a "no accidental damage" boundary, not a security one.
   - **Streaming + tool-calls deferred.** The MVP issues a non-streaming Ollama call when tools are available and only streams the final no-tool answer. Acceptable papercut for now.
6. **Updated [CLAUDE.md](../../CLAUDE.md)** decisions table with a row pointing at ADR 0003.
